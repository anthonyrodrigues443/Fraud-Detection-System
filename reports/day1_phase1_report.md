## Objective
Establish the dataset, understand fraud patterns through thorough EDA, select the primary evaluation metric with domain justification, and build baseline models for credit card fraud detection.

## Research & References
1. **Frontiers in AI (2025)** — "Enhancing credit card fraud detection using traditional and deep learning models with class imbalance mitigation": XGBoost and Random Forest consistently top tabular fraud detection benchmarks. Ensemble methods outperform standalone models.
2. **MLPills Issue #101** — "SMOTE's Limitations in Modern ML": SMOTE blends unrelated fraud types into synthetic examples that never occurred, hurting precision. Cost-sensitive learning (class_weight='balanced', scale_pos_weight) and threshold tuning are preferred alternatives for fraud detection.
3. **Stripe Engineering Blog** — "How ML works for payment fraud detection": Domain features (velocity checks, geographic anomaly, merchant risk scoring, amount deviation) outperform raw transaction features alone. Building the right features matters more than algorithm choice.
4. **Davis & Goadrich (2006)** — "The relationship between PR and ROC curves": For highly imbalanced datasets, AUPRC is more informative than ROC-AUC because it doesn't reward the trivially high true negative rate.
5. **arxiv:2502.00201** — "Year-over-Year Developments in Financial Fraud Detection via Deep Learning: A Systematic Literature Review": LSTM shows sustained growth for sequential fraud data; Transformer-LOF-RF hybrid models show promise for rare pattern detection.

How research influenced today's experiments: Selected AUPRC as primary metric (per Davis & Goadrich), chose a dataset with real feature names to enable domain feature engineering (per Stripe), included cost-sensitive LogReg baseline alongside default (per SMOTE limitations research).

## Primary Metric Selection
**AUPRC (Area Under Precision-Recall Curve)**

**Justification:** For fraud detection with extreme class imbalance (~0.58% fraud rate in our dataset), ROC-AUC is misleadingly optimistic because the true negative rate is always near 1.0. A model that catches 50% of fraud while generating 1% false positives looks great on ROC (AUC ~0.95) but terrible operationally (precision ~23%). AUPRC directly measures the precision-recall tradeoff: can we catch fraud without drowning investigators in false positives?

This is the standard metric used in:
- IEEE-CIS Fraud Detection competition top solutions
- Industry fraud systems (Stripe, PayPal) where precision at high recall drives operational cost
- Published fraud detection benchmarks (Frontiers in AI 2025)

**Secondary metrics:** F1-score, Precision@95%Recall, ROC-AUC, Recall@5%FPR

### Why this dataset?
Unlike the classic ULB Credit Card dataset (28 anonymized PCA components) or IEEE-CIS (431 mostly anonymized features), this dataset has **real, interpretable feature names**: merchant, category, amount, customer location, merchant location, age, gender, timestamps. This enables meaningful domain feature engineering — velocity checks, geographic impossible travel, merchant category risk scoring — which is exactly what production fraud systems use.

**Limitation:** The data is simulated (generated by Sparkov tool), not real transaction data. However, the simulation models realistic fraud patterns and the feature structure mirrors production transaction data.

## Experiments

### Experiment 1.1: Majority Class Baseline
**Hypothesis:** Establishes the floor — always predicting "legitimate" achieves ~99.4% accuracy but zero fraud detection.
**Method:** DummyClassifier(strategy='most_frequent')
**Result:** AUPRC=0.0057, F1=0.0, Recall=0.0, ROC-AUC=0.50
**Interpretation:** Accuracy is meaningless for fraud detection. A model that never detects fraud is useless despite 99.4% accuracy. This proves why AUPRC is the right metric.

### Experiment 1.2: Logistic Regression (default)
**Hypothesis:** Simplest ML baseline; with default threshold, will sacrifice recall for precision on imbalanced data.
**Method:** LogisticRegression(max_iter=1000) on standardized features (17 features)
**Result:** AUPRC=0.3622, F1=0.1653, Precision=0.6807, Recall=0.0941, ROC-AUC=0.8845
**Interpretation:** High precision (68%) but catches only 9.4% of fraud. Default threshold is far too conservative for operational use. The model has learned useful signal (AUPRC 63x above random) but the threshold needs tuning.

### Experiment 1.3: Logistic Regression (cost-sensitive)
**Hypothesis:** class_weight='balanced' inversely weights classes, forcing the model to pay more attention to fraud. Should improve recall at the cost of precision, and improve AUPRC.
**Method:** LogisticRegression(max_iter=1000, class_weight='balanced')
**Result:** AUPRC=0.2484, F1=0.0768, Precision=0.0402, Recall=0.8468, ROC-AUC=0.9411
**Interpretation:** **COUNTERINTUITIVE FINDING:** Cost-sensitive LogReg has WORSE AUPRC (0.2484) than default (0.3622), despite higher recall (85% vs 9%). The precision crash to 4% means 24 false positives per true positive — operationally unusable. ROC-AUC (0.94) is misleadingly high because it rewards the still-excellent TNR. This perfectly illustrates why AUPRC > ROC-AUC for imbalanced data, and validates the literature: class reweighting isn't a free lunch.

### Experiment 1.4: XGBoost Baseline
**Hypothesis:** Tree-based models handle imbalance better with scale_pos_weight. XGBoost should significantly outperform LogReg on tabular data.
**Method:** XGBClassifier(n_estimators=200, max_depth=6, scale_pos_weight=174, eval_metric='aucpr')
**Result:** AUPRC=0.9314, F1=0.5563, Precision=0.3916, Recall=0.9600, ROC-AUC=0.9986, Prec@95%Recall=0.5009
**Interpretation:** XGBoost is DOMINANT — AUPRC 2.6x higher than best LogReg. Catches 96% of fraud with 39% precision (2.5 false positives per true catch). Precision@95%Recall of 50% means even at very high recall, every other flagged transaction is real fraud. This sets a very high bar for Phase 2.

## Key Findings
1. **is_night is the #1 predictor** (XGBoost importance=0.41): Fraud rate at 10pm is 2.82% vs 0.08% at 6am — a 35x difference. Fraudsters operate when victims are asleep and less likely to notice.
2. **Transaction amount is #2** (importance=0.35): Fraud transactions have distinctly different amount distributions than legitimate ones.
3. **Class reweighting HURTS overall precision-recall tradeoff**: LogReg balanced has WORSE AUPRC (0.2484) than default (0.3622). This validates the SMOTE-skepticism literature — simply reweighting isn't the answer.
4. **ROC-AUC is misleading for fraud**: LogReg balanced shows 0.94 ROC-AUC but only 0.25 AUPRC. A metric that says "this is a great model" when it has 4% precision is the wrong metric.
5. **Temporal pattern is the strongest single signal**: Peak fraud hour (22:00) and the is_night feature dominate. This aligns with Stripe's observation that behavioral timing features are among the most discriminative.

## Key Domain Features Engineered (Phase 1)
1. **Temporal:** hour, day_of_week, month, is_weekend, is_night (10pm-5am)
2. **Geographic:** customer-merchant haversine distance (km)
3. **Amount:** log-transformed amount
4. **Demographic:** age (from DOB), gender
5. **Category:** label-encoded merchant category

## Planned Domain Features (Phase 3)
- Velocity: transaction count/amount per card in last 1h/24h/7d
- Amount deviation: z-score of amount vs card's historical mean
- Merchant risk: historical fraud rate per category (target encoding, leakage-safe)
- Geographic impossible travel: speed between consecutive transactions
- Time since last transaction per card
